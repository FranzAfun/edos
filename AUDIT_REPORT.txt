MIGRATION PLAN: localStorage TO REAL DATABASE
----------------------------------------------
Once the backend is set up:
  1. Create the .env file with database credentials (DB_HOST, DB_PORT,
     DB_DATABASE, DB_USER, DB_PASSWORD) pointing to Laragon's MySQL.
  2. Run the schema/migrations to create all tables in the database
     (viewable and manageable via HeidiSQL in Laragon).
  3. Once the database is live and API endpoints are connected,
     ALL localStorage stores will be removed entirely:
     - approvalStore.js    (replaced by Approvals API + DB)
     - budgetStore.js      (replaced by Budget API + DB)
     - complianceStore.js  (replaced by Compliance API + DB)
     - departmentStore.js  (replaced by Department API + DB)
     - kpiStore.js         (replaced by KPI API + DB)
     - notificationStore.js(replaced by Notification API + DB)
     - userStore.js        (replaced by User API + DB)
  4. The httpClient.js will be updated to make real HTTP requests
     to the backend API instead of returning empty mock data.
  5. The createModuleService wrapper stays — only the data source
     changes from localStorage to API calls. No UI rewrites needed.
  6. Seed data currently hardcoded in localStorage stores will be
     moved to database seeders/migrations.

  5. No emojis anywhere in the UI (currently clean - maintain this).

  7. Consistent font weights: use 400 for body, 500 for labels,

SECURITY (when backend is built):

     session cookies — browser sends cookies automatically, so CSRF
     tokens must guard POST/PUT/DELETE routes).
  9. Password complexity requirements (min 12 chars, mixed case, numbers).
  10. Session timeout/expiry (30 min idle, 8 hour absolute).
  11. IP-based rate limiting on login (5 attempts per 15 min).
  12. Two-factor authentication for CEO and FO roles.
  13. API request signing for sensitive financial operations.

ARCHITECTURE:
  14. Add state management (Zustand or TanStack Query) when replacing
      localStorage with real API calls.
  15. API versioning strategy (/api/v1/).
  16. Request/response logging middleware.
  17. Health check endpoint for monitoring.
  18. Database connection pooling.

PERFORMANCE & LIGHTHOUSE OPTIMIZATION (ongoing as we build):

  20. Tree-shake unused Tailwind classes (already handled by Tailwind 4
      purge, but verify production bundle).
  21. Lazy-load heavy components (charts, data tables, rich editors).
  22. Compress and optimize all images (favicon, receipts, evidence).
      Use WebP/AVIF where possible.
  23. Add proper meta tags, semantic HTML, and accessible labels to
      pass Lighthouse Accessibility audit (aim for 90+).
  24. Preconnect/preload critical resources (fonts, API origin).
  25. Enable gzip/brotli compression on the backend server.
  26. Cache static assets with proper Cache-Control headers.
  27. Minimize layout shifts (CLS) — use explicit width/height on
      images, skeleton loaders instead of content pop-in.
  28. Keep JavaScript bundle under 200KB gzipped for fast First
      Contentful Paint (FCP) and Time to Interactive (TTI).
  29. Use font-display: swap on Google Fonts (already in link tag)

  30. Run Lighthouse CI on every build to catch regressions early.

      Best Practices 90+, SEO 90+.
